# bioStudy 결정 기록

이 문서는 주요 제품·기술 결정과 그 이유를 기록한다. 상세 요구사항은
[`product-spec.md`](product-spec.md), 공통 규칙은 [`../CODEX_MASTER.md`](../CODEX_MASTER.md)를 따른다.

| # | 결정 | 이유 |
|---:|---|---|
| 1 | MVP 사용자는 서비스 소유자 한 명이다. | 개인 학습 지속성에 집중하고 계정·권한 복잡도를 피한다. |
| 2 | MVP 학습 상태는 localStorage에 저장한다. | 로그인과 서버 운영 없이 단일 브라우저에서 빠르게 시작할 수 있다. |
| 3 | 학습 콘텐츠는 저장소 내부 JSON 파일로 관리한다. | 순서, 참조, 문제 정답과 승인 상태를 구조적으로 검증하기 쉽다. |
| 4 | 기본 시간대는 `Asia/Seoul`이다. | 오늘 세션, 완료일과 streak의 날짜 경계를 일관되게 계산한다. |
| 5 | Vercel 연결은 Phase 1 구현과 production build 검증 완료 후 진행한다. | 실행 코드가 없는 Phase 0에서 배포 연결을 서두르지 않는다. |
| 6 | AI 피드백은 선택적인 Phase 6 기능이다. | 핵심 학습 흐름을 외부 API와 비용·장애에 의존시키지 않는다. |
| 7 | AI 없이도 일일 복습부터 완료 저장까지 전체 흐름이 작동해야 한다. | 직접 설명은 자기 확인만으로 완결되어야 한다. |
| 8 | 콘텐츠 구조 검증과 과학적 정확성 승인은 별도 절차다. | 스키마 통과만으로 사실의 정확성과 교육 적합성을 보장할 수 없다. |
| 9 | 실제 학습에는 `APPROVED` 콘텐츠만 노출한다. | 검토되지 않은 초안이 학습 자료로 사용되는 것을 막는다. |
| 10 | AI나 Codex가 만든 콘텐츠는 기본적으로 `DRAFT`다. | 사람 검토 전 자동 승인으로 오인되는 것을 방지한다. |
| 11 | Phase 5는 자동 승인 단계가 아니다. | 5A는 DRAFT 카탈로그·검증 체계, 5B는 사람 검수 기반 투입 과정이다. |
| 12 | 상용화, 다중 사용자와 게임화는 MVP 범위 밖이다. | 약 15분의 개인 학습 지속성보다 부차적인 기능 확장을 막는다. |

## Phase 경계 검토 기록

- Phase 3은 버전된 localStorage 상태를 확장하며 DB, ORM 또는 동기화를 도입하지 않는다.
- Phase 5는 120개 구조와 승인 게이트를 마련하지만 콘텐츠 대량 자동 생성·자동 승인을 하지 않는다.
- Phase 6은 선택적 보조 기능이며, AI가 없어도 학습 완료·저장·진도가 작동한다.
- Vercel 연결은 Phase 1 구현 완료 뒤 사용자가 수행한다.
- Phase 7의 백업·복원 전에는 localStorage가 유일한 MVP 상태 저장소다.
